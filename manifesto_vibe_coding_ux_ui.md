# O Manifesto do Design System Hype e Vibe Coding
### Repositório Profundo de Arquitetura Visual, Canvas, Motion Web e UX/UI de Vanguarda

Este documento é a base de conhecimento definitiva para alimentar modelos de IA na construção de plataformas complexas de design visual (como editores online e workspaces colaborativos). Aqui estão as tecnologias de ponta, bibliotecas de componentes e os paradigmas visuais usados pelos engenheiros de design mais avançados do mundo.

## 2. Motores de Renderização: Canvas, WebGL e Vetores
Para construir um editor gráfico ou ambiente espacial online, o DOM padrão do HTML não suporta a performance necessária. Estas são as bibliotecas de baixo nível que a IA precisa dominar para gerar o motor da plataforma.

* **PixiJS ([pixijs.com](https://pixijs.com)):** Motor 2D ultrarrápido usando WebGL. É a escolha definitiva quando se precisa renderizar milhares de objetos simultâneos na tela (como dezenas de pranchetas) mantendo 60fps constantes.
* **Konva.js ([konvajs.org](https://konvajs.org)):** Um framework 2D focado em interatividade de canvas para desktop e mobile. É excelente para criar lógicas complexas de seleção, arrasto (drag & drop), redimensionamento e agrupamento de nós visuais.
* **Fabric.js ([fabricjs.com](https://fabricjs.com)):** Fornece um modelo de objeto interativo em cima da API nativa de Canvas. Muito utilizado para editores de imagem SVG e manipulação de objetos gráficos direto no navegador.
* **Tldraw ([tldraw.dev](https://tldraw.dev)):** Mais do que uma biblioteca, é um motor de quadro branco (whiteboard) colaborativo pronto. A arquitetura de estado deles é a referência máxima atual para construir ferramentas espaciais modernas.

## 3. Motion Web e Ritmo Cinético (Para Direção de Arte Fluida)
Plataformas de alta tecnologia não mudam apenas de estado, elas coreografam a transição. Compreender como traduzir timelines e keyframes complexos para código é vital.

* **Theatre.js ([theatrejs.com](https://theatrejs.com)):** Um divisor de águas. Traz a lógica de keyframing, timelines e interpolação de curvas de Bezier (muito familiares a ambientes de motion graphics e edição de vídeo) direto para o código de interfaces e cenas WebGL.
* **GSAP (GreenSock) ([greensock.com/gsap](https://greensock.com/gsap)):** A biblioteca de animação padrão da web comercial há anos. Extremamente robusta para encadear animações sequenciais complexas em interfaces pesadas.
* **Lenis ([lenis.darkroom.engineering](https://lenis.darkroom.engineering)):** Biblioteca de scroll suave que se tornou padrão em sites premiados, garantindo que a navegação do usuário responda de forma física e com peso real.
* **Rive ([rive.app](https://rive.app)):** O substituto moderno do Lottie. Permite que animações interativas e complexas rodem em tempo real com estado condicional (state machines), ideal para ícones interativos e mascotes dentro de apps.

## 4. O "Graal" da Experiência do Usuário (UX/UI Benchmarks)
Alimente a IA com as lógicas dessas plataformas quando quiser que ela replique os melhores padrões de atalhos, navegação e organização visual.

* **Linear ([linear.app](https://linear.app)):** A principal referência estética e de UX da atualidade. Popularizou o design escuro corporativo (dark mode SaaS), menus de comando (Ctrl+K) instantâneos e feedback visual ultrarrápido.
* **Raycast ([raycast.com](https://raycast.com)):** Foco absoluto na usabilidade por teclado. Se sua plataforma precisa ser usada por power users (como designers), o fluxo de trabalho sem mouse do Raycast é a referência.
* **Vercel v0 ([v0.dev](https://v0.dev)):** O estado da arte em IA gerando interface. Estudar como o v0 estrutura seus outputs ajuda a guiar o seu próprio processo de vibe coding para criar componentes React coerentes e bem estilizados.

## 6. Boas Práticas para Vibe Coding e Desenvolvimento Profissional com IA

Para maximizar a eficiência no fluxo de *vibe coding* (onde desenvolvedor e IAs iteram rapidamente no código e no design), devem ser seguidas as seguintes regras de engenharia:

### 6.1. Segurança de Estado (Zustand & React Context)
- **Desestruturação Segura:** Ao injetar variáveis globais complexas (ex: `exportConfig`), sempre as inclua na desestruturação inicial do hook de estado no topo do componente. Omissões causam quebras de execução (`ReferenceError`) que congelam o renderizador do React.
- **Fidelidade de Configuração:** Mantenha configurações e patches de estado estritamente alinhados com os esquemas JSON originais para evitar inconsistências nos motores de renderização e loops infinitos no HMR.
- **Seletores Atômicos para Scrubbing:** Em interações de alta frequência (como o arrastar da agulha ou a edição de trim em tempo real), utilize seletores Zustand atômicos (ex: `const duration = useEditorStore(state => state.exportConfig.duration)`) em vez de ler o objeto de configuração inteiro. Isso evita re-renderizações em cascata e lags visuais durante o scrubbing.

### 6.2. Higiene de Imports e Componentes Órfãos
- **Refatoração Limpa:** Ao migrar ou centralizar controles (como fizemos ao mover o controle de cores das paletas e a biblioteca para fora do escopo original), certifique-se de fazer um rastreamento completo de referências cruzadas no repositório. Nunca deixe tags e componentes órfãos no HTML (ex: `<ColorManager />` sem import correspondente), pois isso inviabiliza o carregamento dinâmico.
- **Checagem de Linter:** Use e abuse do linter do Vite e logs HMR para identificar avisos de módulos não declarados ou variáveis duplicadas antes de dar a tarefa por encerrada.

### 6.3. Desacoplamento e Hierarquia Visual
- **Centralização de Cena:** Definições fundamentais da cena (Background de Vídeo, Imagem ou Cor Sólida, Aspect Ratio, FPS e Duração) pertencem unicamente a um módulo de **Composição**. Outros módulos (Tipografia, Generativo, Biblioteca) devem apenas ler esses dados sem nunca tentar alterá-los ou sobrescrevê-los.
- **Flexibilidade no Canvas:** O canvas principal deve atuar como uma viewport neutra, renderizando o background por baixo das visualizações ativas. Isso permite edição fluida sob as condições reais em que o arquivo final será exportado.

### 6.4. Loop de Validação Contínua (HMR & Build)
- **HMR Sanity Check:** Certifique-se de que a aplicação se recupera perfeitamente de Fast Refreshings no navegador sem quebrar o lifecycle de plugins terceiros (como timelines do GSAP e animações acionadas por ruído Simplex).
- **Validação de Build:** Rode `npm run build` localmente para atestar que o bundle final está livre de quebras de TypeScript antes de fazer commits na branch principal do projeto.

### 6.5. Estética Bento Grid & Glassmorphism (SaaS Hype Standard)
- **Grid de Bento Consistente:** Organize os painéis utilizando containers Bento estilizados com bordas sutis e fundo translúcido:
  - Bordas: `1px solid var(--color-surface-border)`
  - Fundo: `var(--color-bg-elevated)` ou `var(--color-surface-glass)`
  - Arredondamento: `border-radius: 8px` ou `12px` para consistência tátil.
- **Variação CSS Customizada:** Centralize a identidade de cor nas variáveis nativas do CSS (`--color-accent`, `--canvas-text-primary`, `--canvas-accent`, `--canvas-text-secondary`). Não sobrescreva valores ad-hoc no código para garantir que mudanças de paletas e temas (dark/light) reflitam uniformemente na prancheta de preview e nos arquivos exportados.
- **Micro-interações:** Todos os botões de ação e cards da biblioteca devem possuir transições suaves no estado de `:hover` com feedback visual imediato (leve mudança de opacidade ou cor de borda) e cursores consistentes (`pointer`).

### 6.6. Linha do Tempo Reativa (React Pointer Events)
- **Abandono de Dependências Pesadas:** Evite a importação de bibliotecas externas complexas e volumosas de arrastar e soltar (como `react-beautiful-dnd` ou `dnd-kit`) para implementar a linha de tempo do editor.
- **Pointer Events Nativos:** Implemente o arraste e o ajuste de duração (trim) de tracks e backgrounds utilizando eventos de ponteiro nativos (`onPointerDown`, `onPointerMove`, `onPointerUp`).
- **Escuta no Objeto Window:** O manipulador `pointerdown` deve ser registrado no elemento da track, porém os manipuladores `pointermove` e `pointerup` devem ser registrados dinamicamente no objeto `window` durante o arraste. Isso garante que o usuário não perca o foco ou pare de arrastar caso o ponteiro saia temporariamente do limite físico da track ou do navegador.
- **Touch Action e User Select:** Adicione `touch-action: none` e `user-select: none` nos elementos arrastáveis para desativar gestos nativos do navegador (scroll, zoom) que interferem na precisão da linha de tempo.

### 6.7. Compositing Híbrido no Canvas de Exportação (Performance Trick)
- **O Gargalo do Vídeo:** Elementos de `<video>` renderizados diretamente no DOM causam sérios gargalos de desempenho e erros de cross-origin ou renderização inacabada quando lidos diretamente por bibliotecas de captura como `html-to-image`.
- **Compositing em Duas Camadas:**
  1. Oculte temporariamente o elemento de vídeo de fundo configurando `display: none` antes de iniciar a captura do frame com `html-to-image`.
  2. Capture a camada do DOM sobreposta (Typography, Generative) como uma imagem PNG transparente limpa.
  3. Restaure o vídeo configurando `display: block` para manter a experiência visível para o usuário.
  4. Crie um canvas off-screen, desenhe o frame atual do vídeo (`ctx.drawImage(bgVideo, 0, 0, width, height)`) e desenhe a imagem PNG capturada por cima no mesmo canvas.
  5. Envie este canvas unificado para o encoder. Isto garante renderização com performance máxima, suporte absoluto a vídeos de fundo em 60fps e fidelidade cristalina nas camadas de texto e vetores.

### 6.8. Engenharia de FFmpeg.wasm no Navegador e Prevenção de OOM
- **Codec libx264 e Dimensões Pares:** O codificador `libx264` exige estritamente que a largura e a altura do vídeo de saída sejam números pares. Sempre aplique um filtro de escala do FFmpeg (ex: `-vf scale=trunc(iw/2)*2:trunc(ih/2)*2`) nos argumentos de execução do FFmpeg para evitar falhas silenciosas de codificação e erros de exportação de MP4.
- **Economia de Memória (OOM Avoidance):** Em renderizações de MP4 (onde o canal alfa não é necessário), converta o frame composto no canvas para o formato `image/jpeg` (JPEG com compressão de qualidade ~0.9) antes de empilhar o buffer na memória. JPEGs ocupam até 80% menos memória em trânsito do que PNGs, evitando falhas por estouro de memória (Out-Of-Memory) do navegador ao exportar composições mais longas em resoluções altas (1080p ou superior).
- **MOV para Canal Alpha (Transparência):** Para renderizações com canal de transparência (MOV), utilize frames intermediários PNG (`image/png`) e ajuste o pixel format do FFmpeg para `-pix_fmt rgba` com o codec `-c:v png` (ou codecs de compressão RLE nativos do QuickTime).
- **Descarregamento de Recursos (Memory Leaks):** Libere explicitamente os buffers e revoque URLs de blobs criadas no navegador (`URL.revokeObjectURL`) imediatamente após o uso de cada frame. Certifique-se de chamar a destruição ou limpeza de instâncias do FFmpeg quando necessário para reaver a memória heap do WebAssembly.

### 6.9. Navegação de Câmera Espacial e Zoom/Pan
- **Navegação Multimodal:** O editor visual deve fornecer navegação espacial de prancheta contínua, permitindo ao usuário aproximar-se para ajustes pixel-perfect ou afastar-se para visualizar a composição macro.
- **Orquestração de Eventos:**
  - **Zoom com Zoom-Wheel:** Capturar o evento de `wheel` e verificar se a tecla `Ctrl` ou `Cmd` (`e.ctrlKey || e.metaKey`) está pressionada para manipular o eixo Z.
  - **Pan com Wheel normal:** Se `Ctrl/Cmd` não estiver ativo, os deltas X e Y da roda (`deltaX`, `deltaY`) devem deslocar os eixos X e Y da câmera para emular movimentação de trackpad nativa.
  - **Pointer Capture:** Para gestos de arraste dedicados (ex: barra de espaço segurada ou clique do meio), chame `setPointerCapture` no ponteiro ativo para rastrear movimentos mesmo quando o cursor sai do viewport, liberando-o com `releasePointerCapture` ao soltar.

### 6.10. Normalização de Escala de Controles de Canvas (UI Inverse-Scale & Container Queries)
- **Gizmos e Overlays estáticos no Zoom:** Quando o canvas principal é transformado via CSS `scale()`, todos os filhos herdam o tamanho redimensionado. Para ferramentas de manipulação de vértice (como o `InteractiveGizmo` e barras flutuantes), isso torna as alças difíceis de selecionar ou excessivamente pixeladas.
- **Aplicação da Escala Inversa:** Calcule a escala final aplicada ao canvas e gere seu inverso no escopo global (`--inverse-scale: 1 / finalScale`). Nos componentes que flutuam sobre a prancheta de edição, aplique a escala inversa via transformações de CSS (`transform: scale(var(--inverse-scale))`). Isto compensa a magnificação do canvas e mantém a escala física das ferramentas constante na tela do usuário.
- **Uso de Container Queries (`cqw`):** Para evitar que fontes e efeitos de rastro (Trails) se desloquem ou percam fidelidade ao escalar o viewport ou alterar resoluções, declare o wrapper de prancheta de resolução fixa como um container lógico (`containerType: inline-size`). Use fontes dimensionadas com `cqw` para garantir que o texto seja renderizado de forma 100% proporcional ao container interno em vez do viewport global (`vw` ou `rem`).
